# 🌿 1-3. 브랜치 네이밍 컨벤션

## 🎯 목적
브랜치 이름만 보고도 **무슨 작업을 하는지, 어떤 이슈를 해결하는지 한눈에 알 수 있도록** 합니다.  
모든 작업은 이슈 단위로 관리하며, 브랜치는 이슈를 기준으로 생성합니다.

---

## 🧩 기본 원칙
- 브랜치는 **작업 목적**에 따라 prefix를 사용합니다.  
- 모든 작업은 **이슈 생성 → 브랜치 생성 → 커밋 → PR** 순서로 진행합니다.  
- main 브랜치는 보호되어 있으며, PR 승인 없이 직접 push 금지합니다.

---

## 🌱 브랜치 구조

| 브랜치           | 역할 |
|---------------|------|
| **release**   | 실제 배포용 안정 버전 |
| **develop**   | 개발 통합 브랜치 (테스트 통합용) |
| **feature/**  | 신규 기능 개발 |
| **fix/**      | 버그 수정 |
| **refactor/** | 코드 리팩토링 |
| **hotfix/**   | 긴급 수정 (배포 후 발생한 문제) |

---

## 🧱 브랜치 네이밍 규칙

| 유형 | 규칙 | 예시 |
|------|------|------|
| 기능 추가 | `feature/<이슈번호>-<작업내용>` | `feature/12-add-login` |
| 버그 수정 | `fix/<이슈번호>-<수정내용>` | `fix/23-auth-token-error` |
| 리팩토링 | `refactor/<이슈번호>-<대상모듈>` | `refactor/45-user-service` |
| 긴급 수정 | `hotfix/<문제요약>` | `hotfix/critical-login-error` |
| 배포 준비 | `release/<버전>` | `release/v1.0.0` |

> 💡 `<이슈번호>`는 GitHub Issue 번호를 사용합니다.  
> 브랜치 이름은 **소문자**, **하이픈(-)** 으로 구분하며 3~5단어 이내로 작성합니다.

---

## ⚙️ 브랜치 생성 예시
| 단계  | 명령어                                    | 설명                                                                                                                           |
|-----| -------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------- |
| 1️⃣ | `git checkout develop`                 | 기능 개발의 기준이 되는 **develop 브랜치로 이동**합니다. 새로운 브랜치는 항상 이 브랜치를 기반으로 만들어야 합니다.                                                      |
| 2️⃣ | `git pull origin develop`              | 원격 저장소(`origin`)의 최신 develop 코드를 **내 로컬 브랜치에 동기화**합니다. 팀의 최신 코드 상태를 반영합니다.                                                   |
| 3️⃣ | `git checkout -b feature/12-add-login` | develop을 기준으로 새로운 브랜치 **feature/12-add-login**을 생성하고 이동합니다.<br>→ `feature/`는 기능 개발용, `12`는 이슈 번호, `add-login`은 작업 내용을 의미합니다. |
