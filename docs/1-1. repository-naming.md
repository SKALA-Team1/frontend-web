# 🏷️ 1-1. Repository Naming Convention

## 🎯 목적
레포지토리 이름만 보고도 **역할·도메인·기술 스택을 한눈에 파악**할 수 있도록 하기 위함입니다.  
일관된 네이밍 규칙은 협업과 CI/CD 자동화 시 혼란을 줄입니다.

---

## 📘 기본 규칙

| 항목 | 규칙 | 예시 |
|------|------|------|
| **소문자만 사용** | 대문자, 공백 금지 | `frontend-web` ✅ / `Frontend Web` ❌ |
| **단어 구분은 하이픈(-)** | `_` 대신 `-` 사용 | `backend-api` ✅ |
| **역할 중심 명명** | 코드의 책임과 역할을 표현 | `ai-agent-service`, `infra-deploy` |
| **짧고 명확하게** | 3~4단어 이내 권장 | `data-pipeline-engine` |
| **기술 스택 포함 (선택)** | 필요 시 구체화 | `backend-fastapi`, `frontend-nextjs` |
---

## 🧠 팀 합의 문장
> “모든 레포지토리 이름은 소문자와 하이픈(`-`)을 사용하며,  
> `<역할>-<핵심기능>-<스택(선택)>` 형식을 따른다.  
> 이름은 짧고 명확하게 작성해, 레포의 역할이 즉시 이해되도록 한다.”
