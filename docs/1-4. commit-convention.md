# 🧾 1-4. 커밋 메시지 컨벤션 (Commit Message Convention)

## 🎯 목적
커밋 메시지를 표준화하여 코드 변경 이력만 보고도  
**“무엇이, 왜 바뀌었는지”**를 명확히 이해할 수 있도록 합니다.  
또한 PR 리뷰, CI/CD 자동화, CHANGELOG 자동 생성 등에도 활용됩니다.

---

## 🧩 기본 구조
커밋 메시지는 아래 형식을 따릅니다:
`<type>(<scope>): <subject>`


| 항목 | 설명 | 예시 |
|------|------|------|
| **type** | 변경의 목적 | `feat`, `fix`, `refactor`, `docs`, `test`, `chore` 등 |
| **scope** | 변경된 기능 또는 모듈명 (선택사항) | `auth`, `user`, `api` |
| **subject** | 짧은 요약 (명령형) | `add login API` |

---

## ⚙️ 작성 예시

| 유형 | 커밋 예시 | 설명 |
|------|-------------|------|
| 기능 추가 | `feat(auth): 로그인 기능 추가` | 새로운 기능 개발 |
| 버그 수정 | `fix(user): 비밀번호 검증 로직 수정` | 오류 해결 |
| 리팩토링 | `refactor(core): 설정 파일 구조 개선` | 기능 변경 없는 코드 정리 |
| 문서 수정 | `docs(readme): 실행 방법 추가` | 문서 변경 |
| 테스트 추가 | `test(user): 유저 서비스 단위 테스트 추가` | 테스트 코드 작성 |
| 배포 설정 | `chore(ci): 배포 파이프라인 수정` | 빌드, 환경설정 관련 |

---

## 🧱 커밋 작성 규칙

1. **한 커밋 = 하나의 목적**  
   - 예: “로그인 API + 회원가입 UI” ❌ → 기능별로 나누기

2. **subject는 명령형으로 작성**  
   - `added` ❌ → `add` ✅  
   - `fixed` ❌ → `fix` ✅

3. **길이는 50자 이내, 마침표 금지**  
   - ✅ `feat(api): add user search endpoint`  
   - ❌ `feat(api): added user search endpoint.`

---

## 🧠 상세 설명이 필요한 경우
한 줄 요약 아래에 상세 변경 내용을 작성할 수 있습니다:
```
feat(auth): 로그인 기능 추가
- JWT 토큰 발급 로직 추가
- OAuth 로그인 로직 분리
- 테스트 코드 작성 완료
```

> 첫 줄은 요약, 아래는 세부 변경사항을 bullet로 정리합니다.

---

## 🪄 커밋 타입 요약

| Type | 의미 |
|------|------|
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `refactor` | 코드 구조 개선 (기능 변화 없음) |
| `docs` | 문서 관련 변경 |
| `style` | 코드 포맷팅, 세미콜론 등 스타일 변경 |
| `test` | 테스트 코드 추가 또는 수정 |
| `chore` | 빌드, 설정, 패키지 관련 작업 |
| `perf` | 성능 개선 |
| `ci` | CI/CD 파이프라인 관련 변경 |

---

## 🚫 피해야 할 예시

| 잘못된 메시지 | 문제점 |
|----------------|----------|
| `update file` | 변경 내용 불명확 |
| `fix bug` | 어떤 버그인지 알 수 없음 |
| `commit` | 의미 없음 |
| `modify code` | 목적 불명확 |
| `add login` | type(prefix) 누락 |

---

## ✅ 팀 합의 문장
> “모든 커밋은 `<type>(<scope>): <subject>` 형식을 따르며,  
> type은 `feat`, `fix`, `refactor`, `docs`, `test`, `chore` 중 하나를 사용한다.  
> subject는 명령형으로 작성하고 50자 이내로 유지한다.”
